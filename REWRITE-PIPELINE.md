# REWRITE-PIPELINE.md — 文章改寫品質管線

> 確保每篇文章從研究到發布，品質不會在任何階段掉落。
> 這份文件是 EDITORIAL.md 的「執行手冊」——EDITORIAL 定義「好文章長什麼樣」，Pipeline 定義「怎麼做出好文章」。

---

## 核心問題：為什麼後半段總是掉品質？

**診斷（2026-03-24 實戰觀察）：**

1. **Token 耗盡**：一個 session 要做 8+ 次搜尋 + 讀 EDITORIAL + 寫 150 行中文 + 寫英文版 + build 驗證 = token 不夠，後半段變草稿
2. **沒有中間 checkpoint**：agent 從頭寫到尾，沒人在中間檢查「你後半段有沒有掉品質」
3. **結尾是最後寫的**：人類和 AI 都一樣——到最後精力不夠，結尾變罐頭
4. **富文本被遺忘**：EDITORIAL 寫了 callout / stat block / pull quote 規範，但 agent 到後面忘記用

---

## Pipeline：三階段分離法

### 原則：研究、寫作、驗證分開做

不要在一個 session 裡做所有事。把一篇文章拆成三個獨立步驟，每一步都有明確的輸入和輸出。

```
Stage 1: RESEARCH（研究筆記）
    ↓ 輸出：research-notes.md
Stage 2: WRITE（撰寫全文）
    ↓ 輸出：article.md（中文 + 英文）
Stage 3: VERIFY（品質驗證 + 修正）
    ↓ 輸出：final article + commit
```

---

### Stage 1: RESEARCH（研究，預算 40%）

**目標**：產出一份結構化研究筆記，讓 Stage 2 的寫手「不需要再搜尋」就能寫出好文章。

**必做清單：**

```markdown
## 研究筆記：[文章名]

### 反直覺核心句
「這篇文章要讓讀者驚訝的是 ______」

### 三個讀者會問的問題
1. ...
2. ...
3. ...

### 事實素材庫（每條附來源 URL）
- [事實 1]（來源：[URL]）
- [事實 2]（來源：[URL]）
- ...（至少 8-10 條）

### 故事素材（人、場景、對話、衝突）
- [故事 1]：...
- [故事 2]：...

### 數據素材（每個數字附來源+年份）
- [數據 1]：...（來源：[URL]，[年份]）
- ...

### 挑戰/爭議素材
- [爭議 1]：正方... / 反方...
- ...

### 國際視角（至少 1 個英文來源）
- ...

### 結尾素材（預先準備）
⚠️ 必須在研究階段就找好結尾的故事/畫面/數據。
不要等寫到最後才想結尾。
- 可能的餘韻式結尾：...
- 可能的翻轉式結尾：...

### 參考資料清單（5+ URL）
1. [名稱](URL)（一手/二手）
2. ...
```

**品質門檻**：
- [ ] 反直覺核心句寫出來了？（含矛盾/反差）
- [ ] 至少 8 條事實素材，每條有 URL？
- [ ] 至少 2 個故事素材（有人名、場景）？
- [ ] 至少 1 個爭議/挑戰素材？
- [ ] 結尾素材已準備？（⚠️ 最關鍵的 checkpoint）
- [ ] 5+ 獨立來源，含 2 個一手 + 1 個英文？

**不合格就不進 Stage 2。**

---

### Stage 2: WRITE（撰寫，預算 40%）

**輸入**：Stage 1 的研究筆記 + EDITORIAL.md

**規則**：

1. **先讀 EDITORIAL.md**（每次都讀，不假設記得）
2. **先寫結尾**（⚠️ 最重要的規則）
   - 結尾是品質崩塌的起點。如果先寫結尾，就不會因為 token 不夠而草率
   - 從研究筆記的「結尾素材」中選一個，寫成 3-5 行
   - 結尾寫好後，再從開場往下寫
3. **開場前三句**：必須有具體事實（年份/數字/人名）
4. **每 2-3 段**：檢查有沒有放策展人聲音（callout 或呼吸段）
5. **富文本最低量**：
   - A 級文章：至少 2 種（📝 callout 必選 + stat block/timeline/pull quote 任一）
   - B 級文章：至少 1 個 📝 callout
6. **挑戰編織**：不是最後補一段「然而也面臨挑戰」，是編進故事裡
7. **Wikilink**：至少 2-3 個站內連結

**寫作順序（強制）**：
```
1. 結尾（3-5 行）← 先寫！
2. 開場 + 30 秒概覽
3. 正文（按敘事弧線）
4. 參考資料
5. 回頭檢查富文本數量
```

**品質門檻**：
- [ ] 結尾不是罐頭？（不含「繼續發光」「值得期待」「也是台灣的故事」）
- [ ] 開場前三句有具體事實？
- [ ] 至少 2 個 callout？
- [ ] 至少 1 個 stat block 或 pull quote？
- [ ] 挑戰是編織在故事裡，不是補丁？
- [ ] 每個數據都有來源標註？
- [ ] 120-200 行（A 級）/ 80-120 行（B 級）？

---

### Stage 3: VERIFY（驗證 + 修正，預算 20%）

**五指檢測（從 EDITORIAL.md）**：

1. 🫵 **驚訝點**：指出讀者會說「哦？」的那一句
2. ✌️ **兩個轉折**：文章裡有沒有至少兩個真轉折？
3. 🤟 **策展句**：有沒有至少一句不傳遞資訊、只製造理解的句子？
4. 🖐️ **念結尾**：大聲念出來有沒有餘韻？
5. ✊ **一句話轉述**：能不能用一句話跟朋友說「這篇在講什麼」？

**塑膠掃描**：
- 把後半段（從全文 60% 位置開始）單獨讀一遍
- 標記所有「拿掉後文章不會少任何資訊」的句子
- 每個塑膠句替換成事實或刪除

**自動驗證**：
```bash
bash tools/detect-ai-hollow.sh --json | 檢查分數 ≤ 3
npx astro build | 確認不炸
```

**不合格 → 修正後重跑 Stage 3，不直接 commit。**

## Cron 自動改寫的特殊規則

Cron sub-agent 在單一 session 中執行，無法真正分三個 session。但可以在 prompt 中強制分階段思考：

### Cron Prompt 結構（建議）

```
## Phase 1: 研究（前 5 分鐘）
- 搜尋 8+ 次，整理事實素材
- 寫出反直覺核心句
- 準備結尾素材 ← ⚠️ 在研究階段就要想好結尾

## Phase 2: 寫作（中間 5 分鐘）
- 先讀 EDITORIAL.md
- 先寫結尾 ← 不是最後寫！
- 再寫開場 → 正文
- 確保富文本數量達標

## Phase 3: 驗證（最後 2 分鐘）
- 五指檢測
- detect-ai-hollow 分數 ≤ 3
- build + commit + push
```

### Token 預算分配

| 階段 | 建議佔比 | 常見錯誤 |
|------|---------|---------|
| 研究 | 35-40% | 搜太多、每個結果都 web_fetch 全文 |
| 寫作 | 40-45% | 前半段太細、後半段沒力 |
| 驗證 | 15-20% | 跳過驗證直接 commit |

### 防崩潰機制

**「後半段品質鎖」**：寫到全文 60% 時，停下來自問：
- 我還有足夠 token 寫一個好結尾嗎？
- 如果不夠 → 現在就寫結尾，中間段落可以精簡

**「結尾先行」**：把結尾當成文章最重要的部分。
一篇開場好但結尾爛的文章 = 整篇印象是爛的（峰終定律）。

---

## 品質分級（改寫後）

| 等級 | 條件 | 動作 |
|------|------|------|
| ✅ PASS | hollow ≤ 3 + 五指全過 + 結尾不是罐頭 | commit + push |
| ⚠️ PARTIAL | hollow ≤ 3 但結尾/富文本不足 | 標記待改善，下輪優先 |
| ❌ FAIL | hollow > 3 或有事實錯誤 | 不 commit，回到 Stage 1 |
