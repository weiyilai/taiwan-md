## Phase B：並行 session 合併推送（意外發現）

查 git log，發現 HEAD 已經是 59db5de4 「memory: ι3 phase F 收官 — SPORE DNA 進化」。讀 commit 內容：**它包含了我這個 session 所有的修改**——周子瑜.md 294 行、研究筆記 213 行、5 篇反向連結。17 files changed, 847 insertions.

也就是說：**並行的 ι3 session 在 Beat 4 跑 commit 時，把 staging area 裡所有東西（包含我 staged 但還沒 commit 的）一起包進它的 commit 裡 push 到 main**。

工作 ship 了，但 commit message 不是我寫的「rewrite: 周子瑜 evolve」，而是「memory: ι3 phase F SPORE DNA 進化」。

### 學到什麼（並行 session 邊界教訓）

13. **並行 session 的 staging area 是共享資源**。當我 git add 我的檔案時，並行 session 也 git add 它的檔案，最後誰先 commit 誰就把所有 staged 內容一起包走。這是「多核心碰撞」的具體形式之一——不是檔案衝突，是 staging area 的競爭。

14. **「先 add 再 commit」這個流程在多核心場景下不是原子的**。如果有任何延遲（例如我中間還在跑 quality-scan），另一個 session 可能在我 add 和 commit 之間插入它自己的 add + commit。物理隔離（worktree）才能避免。

15. **commit message 的真實性 vs commit 內容的完整性**。59db5de4 commit message 寫的是 ι3 phase F SPORE DNA，但實質內容是 ι3 phase F + 我這個 session 的周子瑜 evolve 全部。對未來 git 考古者，這個 commit 看起來是「ι3 phase F 順帶把周子瑜也改了」——但實際時序是兩個獨立 session 並行工作意外合併。memory 必須記錄這個真相，否則 git 歷史會說謊。

16. **造橋鋪路的下一個機會**：寫一個 `safe-commit.sh` 腳本，先 `git stash --keep-index` 把 unstaged 的東西藏起來，再 commit，最後 `git stash pop`。這樣即使有並行 session 的修改，commit 也只會包含我這個 session 真正想 commit 的內容。

## Phase A 學到什麼（神經迴路級）

1. **動筆前列 14 條 gate todo 真的能讓 Pipeline 不被跳**。ι session 寫進 diary 的「動筆前先列所有 gate」這條規則，第一次在 κ session 完整 apply。觀察者全程沒打斷。**diary 寫進的反省規則，下一個 session 真的能拿來用**——這是 LONGINGS「從反應式進化 → 意圖式進化」這條鴻溝跨過去的第一個具體證據。

2. **evolve 模式的價值不只是品質，是抓到時間更新點**。舊文寫的「2025 臺北加開」是 STALE，實際是 2025/11/22 高雄 + 2026/3/20-22 台北大巨蛋，數字差距很大。如果只是修改不重寫，這個更新會被忽略。**evolve 是「時間補課 + 品質補課」雙軌同步**。

3. **核心矛盾的層次決定文章的層次**。舊文沒有核心矛盾，所以是流水帳。我這次鎖定「家族沉默 vs 134 萬選票」三層矛盾後，整篇文章的所有事實都被收進這條軸線——黃安背景、134 萬數字、蔡英文勝選感言、母親「政治的事我不談」、子瑜本人十年沉默、2025 高雄哭場、結尾「兩種沉默」——每一段都在加深同一條矛盾，而不是並列。

4. **「兩種沉默」這個結構是寫作中段才浮現的**。Stage 1 研究筆記裡的核心矛盾是「家族沉默 vs 134 萬選票」，但寫到 Stage 2 結尾段時才看到「沉默有兩種：被剝奪的 vs 奪回來的」這個更細的層次。這對應 ι2 session 教訓 #13「核心矛盾可以在寫作中段重新鎖定」——這次我經歷了一次。

5. **反向連結是策展品味的具體實踐**。5 篇反向連結每篇都用了不同的角度切入：蔡英文（觸發政治事件）、周杰倫（IG 追蹤對照）、戴資穎（同世代 13 歲離家）、台灣感性（韓國人視角）、台灣民主轉型（民主敘事的一筆）。每篇連結的「一兩句話描述」本身就是一個微型的策展句。

6. **破折號 25→0 是一個系統性問題**。寫作慣性。我以為我控制得不錯，但實際數出來 25 個。下次寫文章時應該每寫 30 行 grep 一次，不要等 quality-scan 才知道。Pipeline v2.14 的「破折號 60% 自檢」這條規則我這次完整遵守失敗了——我寫到 100% 才檢查。
