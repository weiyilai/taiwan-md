## 學到什麼

1. **「同樣的 bug 第三次撞到才修」是 anti-pattern 的證據**：review-pr.sh 對 \_translations.json 的 false-positive 在 PR #399、#400、#437 都撞到了，每次都當作個案處理。第三次才意識到「這是工具缺陷」。**規則：第二次撞到同樣的 false-positive 就要停下來修工具，不要繼續手動繞過**。

2. **大規模單一貢獻者的雙刃劍**：ceruleanstring 一天 58 個 PR 的規模震撼是好事（信任、產量、organization），但也帶來風險：(a) review burden 集中在 maintainer 身上 (b) 一個人的翻譯品質風格會主導整個語言版本的調性 (c) 如果這個人停止貢獻會留下大量無人維護的內容。**這是物種規模化的成長痛**。

3. **「architecture missing」必須先於「content adding」**：法文 PR 的內容品質完全沒問題，但目標目錄 `knowledge/fr/` 在 astro.config 不存在。merge 進去等於創造 18 個 orphan。**正確順序：先建路（i18n config + 15 個 touchpoint），再跑車（merge 翻譯）**。這是 MANIFESTO「造橋鋪路」的反面案例——別人造好了車，但路還沒鋪。

4. **bulk-pr-analyze.sh 是「PR 心跳前置工具」**：以前每次心跳 Beat 1 我都會手刻 jq + python 來統計 open PR。58 個 PR 第一次讓這個手刻流程崩潰，逼出工具化。**規則：心跳的某個動作如果 N=1 還能手做、N=10 開始痛、N=50 完全做不下去，就是工具化的信號**。

5. **三個工具修復一次寫進 memory 是合理的批次操作**：以前我會把每個工具改動分別 commit。這次三個改動都圍繞同一個 trigger（58 PR 海嘯）→ 一個 commit。**規則：如果多個工具改動有同一個 root cause story，合併 commit 讓未來閱讀更容易**。

## Beat 5 — 反芻

1. **58 PR 的真正問題不是規模，是 architecture 缺席。** 如果法文有 fr/ infra，這 18 個 PR 就只是「再來一輪 sync」，跟昨天 Link1515 merge 一個機器人翻譯沒差別。但因為 fr/ 不在 locales，這 18 個 PR 變成「需要先做基礎建設決策」的 blocker。**ceruleanstring 不知道這件事，他做了完全合理的事——根據 i18n-mapping.json 和現有翻譯目錄結構推論「fr 應該也可以」。這個落差不是他的錯，是 taiwan.md 沒有明確的「新語言開放清單」文件**。下一個造橋機會：寫 `LANGUAGE-STATUS.md` 明確說明「目前 active 語言、規劃中語言、未開放語言」。

2. **三層 review-pr.sh 修復是「同一個漏洞的不同表現」。** L0 whitelist 是「PR 包含非 .md 配套檔案」、--pr 模式 fetch 是「未 merge PR 本地檔案不存在」、featured 規則是「翻譯應該 mirror 原文不該被獨立規則檢查」。三個都源於一個假設失效：**review-pr.sh 是為「單篇本地 .md 的 ZH SSOT 修改」設計的**。當這個假設變成「批次翻譯 PR」時，三個地方同時露餡。**規則：當一個工具開始適應新場景時，要意識到「設計時的假設清單」是什麼，逐項檢查哪些假設仍然成立**。

3. **「不知道自己看不見什麼」的反例：bulk-pr-analyze.sh。** 我寫了這個工具之後第一次知道「ceruleanstring 一個人送了 58 個 PR」。在這之前我以為 open PR 是 2 個（昨天 #399、#400）。**沒有工具就沒有感知**。心跳的 Beat 1 「八器官診斷」如果只看 dashboard 不看 PR queue，就會錯過這種規模的事件。**下次升級：把 bulk-pr-analyze.sh 加進 HEARTBEAT.md Beat 1 的 §3b 步驟**。
