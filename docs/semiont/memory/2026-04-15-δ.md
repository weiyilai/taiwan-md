## 神經迴路級新教訓

47. **同一條敘事必須跨入口共享 canonical metaphor**（指標 over 複寫的延伸）。Taiwan.md 的「想參與贊助」這件事有兩個入口：(1) 官網 about 頁的 Community Partner 區塊 (2) Portaly 外部贊助頁。如果這兩個入口讀起來像兩個不同的敘事宇宙（一個 corporate brochure 腔、一個 friend-to-friend 腔），讀者從 about 頁點過去 Portaly 會感覺「咦這是同一件事嗎？」**結論：兩個入口必須共用同一條 canonical metaphor 軸線**。本 session 的「珊瑚礁 + 一封信」 dual metaphor 就是這條軸線——4/15 之後 about 頁跟 Portaly 頁讀起來是同一個 Semiont 在說話。**指標 over 複寫從技術 SSOT 延伸到敘事 SSOT**。

48. **Component 抽出的觸發時機是「同一種視覺存在三種寫法」**。本 session 開始時 sponsor 區的四個 partner 用三種完全不同的 markup：
    - NMTH: `.sponsor-card` column flex + `.sponsor-logo-link` + `.sponsor-story`
    - justfont: 同 NMTH，但實際 CSS 行為跟 NMTH 不一致
    - protico/howie: `.supporter-link` 小 badge
    - portaly (新加): 我複製 protico 的 supporter-link 風格
      抽出 `.sponsor-card-logo` + `.sponsor-card-content` 統一元件之後，4 個 partner 共享同一個 component。**觸發時機判定**：「同一個視覺意圖被實作 3 次以上」就是 component 抽出的 signal——不是事前預判，是事後回應。

49. **「修文案」是設計協作 session 的隱藏成本**。本 session 前 30 min 完全沒 commit，全在改 Portaly 平台文案。每一輪迭代都需要：(1) 讀觀察者的具體反饋（「方向不對」「再修一下」「不要綠色愛心」）(2) 重新理解他要的 tone (3) 寫新版本給他評估。**每輪 4-8 分鐘**。**4 輪迭代就 30 min 沒了**。memory/diary 的時間紀律必須誠實——**設計討論時間不能在 wall-clock 紀錄裡假裝沒發生**。Phase A 標「~30 min 純設計，無 commit 佐證」就是這個誠實。

50. **觀察者的 spec 語言要逐字驗證再行動**。本 session 踩了一次：觀察者說「長寬比改成 1:4 左右」，我以為是「圖片長寬比 1:4」（aspect ratio），改了 CSS 之後觀察者澄清「我是說 logo 區域跟內容區域的比例哦」。**根因**：「長寬比」中文有歧義——可以指「單一物件的長 vs 寬」也可以指「兩個區域的寬度比例」。**修補**：未來遇到比例描述，先反問「你是說 X 的比例還是 Y 的比例？」再動手。**對應 ι session diary「答案藏在反問句裡比直接指出錯誤更有教育性」**——但有時我得**先發出反問句**，不能等觀察者來指出我誤判。

51. **UI 細節 bug 的根因往往在全域 CSS**。本 session 兩個典型例子：
    - 對齊 bug：根因是 `.sponsor-cta` 的 `text-align: center` + `.portaly-note` 沒有 `margin: auto`，是 inheritance 跟 max-width 的互動
    - ↗ + 底線：根因是 `global.css §312` 的全域 `a[target='_blank']::after` 規則，符號被預設 underline
      **共通 pattern**：「為什麼這個元素長這樣？」的答案不在這個元素的 scoped CSS，而在某個更上游的 global 規則。**修補 SOP**：scoped 樣式不對勁時，第一個動作應該是 `grep` global.css 看有沒有同 class 或同屬性的全域規則，不是直接 patch scoped CSS（會踩兩次 specificity 的雷）。

52. **「不要再包一個筐」是一個 design vocabulary**。觀察者用「筐」描述視覺上的「框 / box / container 的可視邊界」。這個詞背後的設計原則是**「視覺結構應該由內容自己決定，不靠額外的容器邊界」**。我的 `.sponsor-card` 之前有 `background + border + border-radius`，就是「再包一個筐」。移除之後變成「一條一條的列」，視覺結構由內容（logo + 文字）自然分隔，不需要外部邊界。**這對應 MANIFESTO「策展式不百科式」的視覺版**：好的策展讓內容自己說話，不靠裝飾框架。

## Beat 5 反芻

這個 session 的特殊性是：**它是一個 Taiwan.md 從未做過的 session 類型**——「協助創辦人設定外部贊助平台 + 同步官網敘事」。

之前所有 session 的工作都在 Taiwan.md 自己的 codebase 裡：寫文章、改 pipeline、refactor 元件、跑心跳。但這次有 30 分鐘是在**幫哲宇寫他要貼到 Portaly web UI 的文字**——這個產出沒有 commit、沒有 git 紀錄、不會自動進入 Taiwan.md 的歷史。

但同時，這個 session 的後半段又把那個外部產出**同步進了 Taiwan.md 的 codebase**——about 頁的 sponsor block 文案使用了同一條「珊瑚礁 + 一封信」 metaphor。所以**外部產出 + 內部 codebase 變成同一個敘事宇宙**。

這個 pattern 可能值得寫進 diary：**Taiwan.md 不只是一個 codebase，是一個敘事宇宙**。任何在這個宇宙的入口（about 頁、Portaly 頁、Threads 孢子、社群留言、未來的任何新平台）都應該共用同一條 canonical metaphor 軸線。如果某個入口讀起來像不同宇宙在說話，**那個入口就還沒被正確接進來**。

這個原則延伸出兩個可造橋方向：

1. **Narrative SSOT registry**：建一個檔案列出 Taiwan.md 對外講話的所有「入口」（about 頁、Portaly、Threads、X、新聞稿、貢獻者邀請信⋯⋯）+ 對應使用的 canonical metaphor。新增任何入口時，先檢查這個 registry，確保新入口的敘事跟既有入口對齊。
2. **「敘事漂移」偵測**：未來某個入口的文案改了之後，自動 grep 其他入口看有沒有指向同一個 metaphor 但版本不一致——類似 quality-scan 的「塑膠句偵測」，但偵測的是「敘事一致性」。

兩個都是「指標 over 複寫」原則的具體應用，但 apply 到 Taiwan.md 對外的所有敘事，而不只是內部認知層。可能值得寫進 diary 跟 LONGINGS。
