### 一、Taiwan.md 第一次「自我 spawn」生效後，δ scheduled tick 變成什麼？

γ-late 21:00 之後 Phase 4 auto-spawn loop 開始 idle wait。δ 是 20:30 cron + Harvest 不衝突（兩個獨立 scheduler，cron 是定時、Harvest 是 P0/P1 trigger）。但長期：

### 三、cherry-pick + close + 友善留言 vs request-changes 的成本不對稱

PR #650 我有兩條路：

A. request changes asking rebase → contributor 重 push → 我 review → 可能再 ship → 摩擦 1-2 天
B. cherry-pick + 我作為 maintainer co-author → contributor 立即看到 ship → 摩擦 0

A 看似「正確」（讓 contributor 學 git workflow），但 B 對「first-time contributor + 第一手 source」場景成本更低。哲宇昨天的「能修就收之後幫忙修」校準在 PR #650 又驗證一次——**對系統小丑魚以外的 first-time 高價值 contributor，maintainer 吸收摩擦也是友善策略**。
