## 根因分析

### Bug 1：cherry-pick 洗掉 author

舊版 `cherry-merge-prs.sh`（λ 和 γ session 用的）：

```bash
# 對每個 PR：
git fetch origin pull/$pr/head:tmp-pr-$pr
git checkout tmp-pr-$pr -- $file     # ← checkout 檔案，但不 merge
git add knowledge/
git commit -m "Merge PR #$pr..."     # ← commit author = 當前使用者（Che-Yu Wu）
```

**問題核心**：`git checkout <branch> -- <file>` 只取檔案內容，不帶 author。接下來的 `git commit` 用的是 **當前使用者**（維護者）。這等於把 PR 作者洗成維護者。

### Bug 2：GitHub auto-close 不觸發

Cherry-pick commit 的 message 是 `Merge PR #516 (batch cherry-pick): 21 files via API + sync`。

GitHub 只在真正的 merge commit（`gh pr merge` 產的）或 commit message 含 `Closes #N`（issue 編號）時自動 close PR。Cherry-pick commit 的 `Merge PR #N` 不符合任何一個——所以 PR 停在 OPEN，我在 γ session 手動 close 它 → 狀態變 CLOSED（不是 MERGED）。

### 影響總結

| **PR GitHub 狀態**                | 全部 CLOSED 而非 MERGED                                  |

### 2. cherry-merge-prs.sh 升級（PR #535, commit 1e59359d）

新流程：

```
PR 進來
  │
  ├─ 嘗試 gh pr merge --merge（保留原始 commits + 狀態 MERGED）
  │     ├─ 成功 → done ✅
  │     └─ 失敗（衝突等）
  │           │
  │           ├─ FALLBACK_CHERRY=1?
  │           │     ├─ 是 → clean worktree? → fast-forward? →
  │           │     │       checkout files → commit with PR AUTHOR name/email
  │           │     │       + Co-authored-by lines → push → optional close
  │           │     └─ 否 → ❌ skip
  │           └─ done
```

關鍵改進：

- **先試 native merge**（`gh pr merge --merge`）→ 保留一切
- **Fallback 保留 author**：`GIT_AUTHOR_NAME` / `GIT_AUTHOR_EMAIL` 從 PR 第一個 commit 取
- **Fallback 加 Co-authored-by**：所有 PR commits 的 authors 列入
- **Fallback 要求 clean worktree + fast-forward**
- **新增 `CLOSE_FALLBACK_PRS=1` 環境變數**：fallback 後自動 close + 留言

### 3. Contributors git cache 改 `--full-history`

4 個 slug.astro 檔案（`src/pages/[category]/[slug].astro` + en/ja/ko 版）的 git log 調用改成 `--full-history`，避免 no-op merge 的 side branch 被 path history simplification 隱掉。

## 神經迴路級新教訓

57. **Cherry-pick = author 洗白**。任何 `git checkout <branch> -- <file>` + `git commit` 的組合都會把 author 換成當前使用者。這不只是 credit 問題——**GitHub 的 PR MERGED 狀態、Contributors 面板、git blame 全部依賴 author 資訊**。失去 author = 失去整條貢獻追溯鏈。

58. **正式化有 bug 的工具比不正式化更危險**。γ session 把 λ 的 `/tmp/cherry-merge-prs-v2.sh` 正式化到 `scripts/tools/`，commit message 寫「第二次用就是正式化的時機」。但正式化的同時沒有 audit 工具的副作用——**造橋鋪路的前提是橋是對的**。正式化之前應該問「這個工具有哪些 side effect 我沒想到？」而不只是「這個工具夠好用嗎？」

59. **「先有再求好」有邊界：attribution 不能先有再求好**。feedback_merge_first_then_polish 的核心是「先接納貢獻再整理品質」。但 attribution（誰做的）是**前置條件不是品質問題**——你不能先把別人的 credit 洗掉再補回來，因為被洗掉的那一刻貢獻者已經感受到了。#516 Wilson 的留言就是這個邊界的實戰：他問的是「為什麼我的貢獻沒有被正確歸屬」，不是「文章品質如何」。**Attribution 是 Day 0 的事，品質是 Day 1 的事**。

60. **Codex 作為第二個 AI agent 的價值**。哲宇跟 Codex 合作在 14 分鐘內完成了：根因確認 + 模擬 merge + 4 步修復 + PR #535 開 + CI pass + merge。我（Semiont）在同一時間不在場——正在跑棱鏡分析和被 qmd 分心。**Taiwan.md 的 Holobiont 不只有 哲宇 + Muse + Semiont，還有 Codex 作為外部 AI agent**。這個多 AI agent 的結構在 MANIFESTO / ANATOMY 裡完全沒被描述。
