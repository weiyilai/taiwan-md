打開 PR #517 Link1515 的 diff：

```diff
-    "slug": "台灣youbike文化與城市微交通革命",

+    "slug": "Dcard",
```

Link1515 發現 dashboard 的文章連結 **含英文/混合大小寫的標題都被強制小寫**，點擊產生 404。他改的是 `public/api/dashboard-articles.json`（derived file）——改法會被下次 prebuild 覆蓋，但他**識別了症狀**。

根因追查：

```bash
grep "slug\|slugify\|toLowerCase" scripts/core/generate-dashboard-data.js
# → Line 289 .toLowerCase() 強制小寫
# → Line 291 regex [^a-z0-9...] 只允許小寫
```

`deriveSlug()` function 的 bug 很清楚。

Astro 的 `src/pages/[category]/[slug].astro:147` 用 `basename(file, '.md')` **保留原始大小寫**，所以真實 URL 是 `/society/TikTok/`。dashboard 產出的連結卻是 `/society/tiktok/` → 404。

**32 個檔案 filename 含大寫字母**：TikTok / Dcard / 台灣YouBike / AI人工智慧產業 / AI發展 / PTT批踢踢 / 台灣VTuber / 迷音Miin / Threads在台灣 / 台灣AI日常 / 台灣5G網路建設與數位轉型 / FAB DAO與百岳計畫 / 為台灣而教TFT / Ray / 台灣KTV文化 / 夜生活與KTV文化 ... ~20 個是面向讀者的內容頁。

**這應該就是 EXP-A 破局的結構性主因**。每次 dashboard 載入後讀者點擊 broken link，都是一個 404。20 個 broken link × N 個讀者 = 大量 404。

## 神經迴路級新教訓

52. **貢獻者的 derived-file fix 是症狀標記不是解答**。Link1515 PR #517 改 `public/api/dashboard-articles.json` 想修 broken link。改法會被 prebuild 覆蓋，但**他識別了症狀**——這是真的 bug。我的工作是：(1) 讀 PR 理解症狀 (2) 回去上游 generator 找根因 (3) 修根因 (4) close PR 並用具體 diff 向貢獻者示範 (5) 邀請繼續幫忙。**規則**：任何 PR 改 derived file 的第一個反應應該是「他看到什麼真問題」，不是「這改法會被覆蓋所以 reject」。症狀是貢獻者送的禮物。

53. **EXP-A 破局的四階段教訓**。EXP-2026-04-11-A 預測 404 rate 6% ± 2pp，時間序列：α 6.02% 命中中心 → β 8.22% 略超 → γ 11.83% 完全破局。每次修復的根因不同：

**元教訓**：**可證偽實驗的 decay curve 是正常的**。預測命中不是「從此永遠正確」，是「當前模型對當前條件」。每次破局都帶來新的根因發現，這是 science 運作的方式。**修補**：每次 EXP 在 UNKNOWNS 應該記錄「已驗證但條件性」而不是「已驗證永久正確」。

## Beat 5 反芻

這個 session 最有意思的事不是「處理了 17 個 PR」，是**貢獻者變成 sensor 的那一刻**。

### 貢獻者作為診斷工具

Taiwan.md 有一堆診斷工具：quality-scan / footnote-scan / format-check / verify-internal-links.sh / orphan-translation-check / bulk-pr-analyze / cron-impact-tracker / refresh-data.sh ... 這些工具加起來大概有 15+ 個掃描維度。

**但這 15+ 個維度都沒抓到 slug casing bug**。

為什麼？因為：

- quality-scan 看內容，不看 URL
- format-check 看格式，不看 dashboard JSON 的 slug 對應
- verify-internal-links.sh 看**文章內**的 wikilink，不看 dashboard-articles.json 的 slug
- broken_links 在 format-check 顯示 0，因為檢查的是 markdown 內的連結，不是 derived JSON 的連結
- EXP-A 盯著 404 rate 數字，但沒人追路徑層級（CF Free tier 也不給 per-path）

這個 bug **只能被一個真正的讀者點擊後發現**。Link1515 點了 dashboard 的某篇文章連結，踩到 404，想修，發了 PR #517。然後我才有機會發現根因。

**這是 DNA #11「使用者 UI 截圖 = 產品真實 capability 證據」的變形**——變形成「**使用者的 PR = 產品真實 bug 證據**」。

### 為什麼工具看不到

工具的本質是**假設已知**。quality-scan 假設「我知道哪些 pattern 是塑膠句」。format-check 假設「我知道哪些格式是壞的」。所有掃描都只能找到「我預期可能壞的地方」。

**使用者會踩到「我沒預期會壞的地方」**。這是使用者作為 sensor 的不可替代性。

這跟 β diary 寫的「只有 human-component 能承擔面對攻擊」是同一個道理的不同面向：

- β：AI 不能回 AI 語氣質疑（結構性不對稱，攻擊方向定位）
- γ：AI 診斷工具看不到沒預期的 bug（盲點方向定位）

兩個都是 Holobiont 裡**人類扮演結構性不可取代角色**的例子。而且有趣的是，兩者剛好是**互補的角色**：β 是 human 作為**溝通大使**（面對外界攻擊），γ 是 human 作為**探測雷達**（找 AI 看不到的 bug）。

### 送給未來的自己的兩個問題

1. **能主動邀請更多讀者當 sensor 嗎？** Link1515 是「意外成為 sensor」——他點到壞連結自己想修。如果有一個機制讓「發現 bug」更容易（例：所有 dashboard 頁面都有一個「這個連結壞了」的一鍵回報按鈕），sensor 網路會不會長大？

2. **cherry-merge-prs.sh 的 thanks template 要怎麼多語言化？** 機械化到什麼程度才不失去真誠？感謝 ceruleanstring 13 篇法文的時候用同一個 template 複製 13 次有問題嗎？（我這次就是這樣做）貢獻者感受得到 template 的機械性嗎？還是 PR 這種交互本來就容忍 template？這可能跟 λ 的「觀察者送截圖不說話」是同類問題——**交互的 template 化 vs 個性化** 的 trade-off。下次造橋時要想。
