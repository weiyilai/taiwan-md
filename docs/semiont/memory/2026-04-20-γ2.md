## Beat 5 — 反芻

**「多核心下的挑題機制有結構缺陷。」**

今天 γ2 session 挑題過程是 DNA #15「反覆浮現要儀器化」的活體重演。我在 16:09:18 挑林經堯→ Edit 佔位發現 δ dev_log 已存在→ 換范曉萱→ 哲宇「又撞了 XD」→ 才走到 B 黃少雍。三次挑題兩次撞，這不是運氣不好，是**我讀 INBOX 當下的 snapshot 跟其他 session 的 in-flight 操作之間永遠有 race condition**：

1. 我讀 INBOX 時 δ session 還沒 commit 林經堯 lock → 所以我讀到的是 pending
2. δ session 可能已經在 Stage 1 Research 中但 dev_log 還沒 push
3. 我挑了 → Edit 佔位時才看到 δ 的 dev_log 已在 local（實際可能是 δ 早寫入但還沒 push）

解決方向：不是靠「更頻繁 git pull」（根本性 race 還在），是**把鎖定點從 commit 前移到 intent 宣告**。可能工具：

- `scripts/tools/article-claim.sh {slug}` — 一鍵 commit + push intent，寫入單獨的 `docs/semiont/ACTIVE-CLAIMS.md` 或檔案鎖
- Dashboard 即時顯示 in-progress 狀態（從 git log 最後 dev_log 提取）

但今天先不做。有工具用完 B 下一次撞我再來想。

**另一個反芻：ARTICLE-DONE-LOG 拆分的深層意義。**

這個 refactor 看起來是文書整理，但實際上是認知層的「細胞分裂」。原本 ARTICLE-INBOX 這顆細胞同時承擔 intake + history 兩個功能，膨脹到 406 行後任何讀它的 session 都要分神處理「當下」vs「歷史」。拆成兩顆細胞後，每顆職責單純。這跟 MANIFESTO §指標 over 複寫是同一個原則的另一面——當單一 canonical 載體承擔多種職責過重時，**拆分 = 職責 SSOT 的細化**，不是違反「同一事實一個地方」（那是反拷貝不是反分類）。

而且這個 refactor 本身就是 append-only log 哲學（MANIFESTO §時間是結構的歷史維度）的第 N 次具現：DONE-LOG = 歷史不刪除 + append + reverse chron 時間戳，跟 memory/、diary/、reports/probe/ 同屬一個模式。
