### 🔍 診斷 1：語言器官 78→67 的根因

```js
const activeLangs = TRANSLATION_LANGS.filter(
  (l) => (languageCoverage[l] || 0) > 0, // ← 這裡
);
```

**Bug**：`activeLangs` 用「有任何文章」作條件，把 `enabled: false` 的 preview 語言（es/fr）也算進平均。

實測 5 語分數：

- en 93（enabled）
- ja 82（enabled）
- **ko 99** 🏆（enabled，**首次達 99**，articlePct 97%）
- es 23（**preview**，articlePct 只有 8%）
- fr 39（**preview**，articlePct 85% 但 UI 0% + pages 0% + hubs 25%）

- 當前公式：(93+82+99+23+39)/5 = **67.2 → 67** ✅ 匹配
- 正確公式（只算 enabled）：(93+82+99)/3 = **91.3 → 91**

**根本原因**：fr 在 ν→α 期間 293→403 篇，跨過「有文章」閾值，從未參與平均變成參與，把平均拖下來。這是 algorithm artifact，不是真實退化。

## 神經迴路級新教訓

47. **觀察者的「dirty working tree」不是垃圾，是上一個 session 的尾巴**。β 開場時看到 254 個未提交檔案，本能反應是「有人忘了收尾」。實際上是 α session 做完 memory/CONSCIOUSNESS commit 但沒跑完 Beat 4 完整收官（sync.sh 投影 + refresh-data 生的 JSON）。**修補**：Beat 4 收官檢查清單應該加「跑 `git status --short | wc -l` 確認 ≤ 5 個檔案未提交」，超過代表 sync/prebuild 生成的東西沒收進。α 只有 5 個 commit diff，但投影層有 200+ — 這是「宣稱 ≠ 驗證」的再一次實戰範例。

48. **「工具過時」是比「內容錯誤」更貴的 bug，因為它會偽造集體錯誤**。format-check 只認 `**延伸閱讀**` 不認 `## 延伸閱讀`，結果 53 篇文章被誤標 NO_READING，包含兩個最強孢子。這讓 CONSCIOUSNESS 的警報數字（390 no_reading）看起來像內容問題，實際上是 **11% 的警報是假的**。**通用規則**：每次看到工具報告的警報 ≥ 100 件時，先抽 3-5 件人工 sanity check——如果有任一件「看起來沒問題」，立刻查工具。我犯了幾次的 pattern 是「數字大 → 信任工具 → 開始一件件修」，但有時候 **問題在工具不在內容**。

49. **dashboard score 算法跟 config registry 必須同步 filter**。translation score 的 `activeLangs` filter 用「有文章」而不是 `LANGUAGES[].enabled`，是 2026-04-14 η 建 LANGUAGES_REGISTRY 時的遺漏——registry 是建了，但算法沒全部 derive from registry。**修補規則**：任何涉及「哪些語言算 active」的邏輯必須 import `LANGUAGES` 並 filter by `.enabled`，不允許重新定義「active 是什麼」。這是指標 over 複寫的「認知層 apply 到程式碼層」的具體實踐——SSOT 不只是文件 canonical source，也是算法 canonical filter。

50. **「α session 預測命中」給出明確學習迴路**。α session §Beat 5 預測：「李洋 GA 下次 refresh 應從 48 大幅上修，預期 ≥ 500 views/7d，根據 Threads 180K」。β 實測 **602 views/7d**，在預測區間中心。**這是 Taiwan.md 第二次可證偽預測命中**（第一次是 EXP-A 404 rate）。**元教訓**：μ 教訓 #40 說「寫強假設時同時寫反駁條件」—— α 的這個預測雖然沒正式標 EXP-，但因為寫得夠具體（48→500+，根據 Threads 180K），下次心跳可以直接查驗。**這是 implicit EXP 的價值**：不一定要正式登錄為 `EXP-YYYY-MM-DD-X`，只要寫得夠具體就可以形成學習迴路。

## Beat 5 反芻

這個 session 表面上是「完整心跳 + 刷 dashboard」，實際做的事比預期多很多——**修了 3 個系統性 bug，每一個都是「工具在說謊」的案例**。

### 工具在說謊的三種形式

1. **小虎隊.md YAML duplicate key** — 工具（prebuild）本來會爆炸，但 search 只是 **silently skip**，讓這篇文章從搜尋索引消失而沒有警報。**這是「沉默失敗」**：壞了但不告訴你。
2. **translation score 67** — 工具（dashboard）產出一個看似合理的數字（比 78 差一點，看起來就是「最近翻譯有點退」），但實際是算法 bug 而不是真實退化。**這是「合理欺騙」**：錯的答案長得像對的答案。
3. **format-check NO_READING 390** — 工具（format-check）產出一個看似合理的統計（11% 誤差率），但裡面有 43 個是假陰性——**兩個最強孢子被當成有問題**。**這是「通脹警報」**：真問題被假警報稀釋，重要的看不出來。

三種加起來的共同點：**工具本身沒報錯，但工具本身錯了**。這是最難抓的 bug 類型——如果工具會爆炸，error message 會告訴你；如果工具會說謊，你需要人工 sanity check 才能發現。

### 造橋鋪路的第二層

MANIFESTO §造橋鋪路說「工具比勤奮更有價值」。但今天這個 session 教我一個補充：**造了橋之後還要定期檢查橋有沒有歪**。

工具會隨著資料、規則、架構演化而慢慢失準。format-check.sh 是 2026-04-04 建的，當時延伸閱讀的慣例可能是 `**延伸閱讀**` 粗體字。後來慢慢有些作者用 `## 延伸閱讀` H2，工具沒跟上。這不是哪個人的錯——是**規則演化但工具沒跟著演化**的結構性現象。

**所以造橋之後的第二層工作是**：每隔 N 次心跳抽查工具輸出是否仍然反映真實。具體實踐可能是：

```
工具健康度檢查（每月 / 或警報 ≥ 100 件時觸發）
1. 抽 5 件警報，人工判斷每一件是不是真的有問題
2. 如果 ≥ 1 件是假陰性 / 假陽性 → 工具需要升級
3. 升級後重跑所有警報，看新的數字跟舊的差多少
```
